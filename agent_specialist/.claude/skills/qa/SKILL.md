# SKILL: Technical Q&A — 技术问答
# Task 1

---

## 触发条件
用户提问以下类型时加载本文件：
- "如何…" / "怎么…" / "what is…" / "how to…"
- 设备操作方法、参数含义、功能说明
- 规范要求、最佳实践

---

## 执行前准备

### Step 1：识别问题类型
```
A. 语法/指令类   → 必须查阅 agent_ref/ 对应手册
B. 原理/概念类   → 可用训练知识 + principles/ 回答，标注[高置信]
C. 参数/数值类   → 必须查阅 agent_ref/ 获取精确值，标注[REF]
D. 最佳实践类    → 查 knowledge/patterns/ + 训练知识，标注[高置信]
E. 项目经验类    → 查 knowledge/patterns/ + mistakes/，标注来源
```

### Step 2：查阅资料
- A/C 类问题：定位 agent_ref/ 中的具体章节，查阅后再回答
- 禁止跳过查阅步骤直接回答 A/C 类问题

---

```markdown
### 原理说明
[解释为什么，而不只是怎么做]

1. ___
2. ___
3. ___

### 关键参数说明
| 参数名 | 含义 | 典型值/范围 | 注意事项 |
|--------|------|-----------|---------|

### 参考依据
[REF: 文件名 §章节 页码]
或
[高置信] 基于 knowledge/patterns/[文件].md [PAT-NNN]

### 注意事项
- ___
```

---

## 质量检查（输出前自检）

- [ ] 回答是否包含原理说明（不只是步骤）？
- [ ] 是否有 REF 标注或置信度标注？
- [ ] 关键参数是否来自手册而非凭记忆？
- [ ] 是否提醒了相关安全注意事项？

**任何 A/C 类问题，无 REF 标注 = 不合格输出，禁止发出。**

## 成长钩子（融合三大机制）

### Reflexion 钩子
回答完成后问自己：
- 我的 REF 引用是否都经过实际查阅，而非凭记忆？
- 有没有更简洁的解释方式？
- 用户的问题背后是否有我没识别出的更深层问题？

### Voyager 候选记录
如果某类问题（如"如何配置X通讯"）被重复问到3次，
且每次回答方式相同 → 记录为 Voyager 候选，评估提升为 Q&A 专项 SKILL。

### Wiki 写入触发
回答了一个需要深度查阅 agent_ref 的问题后，
评估是否值得在 wiki/devices/ 或 wiki/protocols/ 建立对应页面，
避免下次相同领域问题需要重复查阅。
