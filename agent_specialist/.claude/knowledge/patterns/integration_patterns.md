# 系统集成判断规则库
# 适用：Task 7（多设备系统集成）

---

## [PAT-030] 协议选型决策规则
**适用场景**：多设备集成，需要选择通信协议
**判断规则**：
  - 需要实时同步运动控制（<1ms）→ EtherCAT
  - Siemens PLC 为主控 → PROFINET RT/IRT
  - Rockwell/OMRON 生态 → EtherNet/IP
  - 混合设备，非实时 → Modbus TCP（简单）或 OPC-UA（复杂）
  - 传感器层智能化 → IO-Link
  必须给出选型理由，不可只给结论
**例外条件**：客户已指定协议时，按客户要求，但需说明潜在限制
**置信度**：高
**来源任务**：框架初始化
---

## [PAT-031] 架构先于细节原则
**适用场景**：所有系统集成设计
**判断规则**：输出顺序必须是：①系统架构图（设备列表+连接关系）→ ②通信架构（协议+数据流）→ ③接口定义表（信号清单）→ ④程序实现；禁止跳过架构直接写代码
**例外条件**：用户明确只需要某一层的内容时，可跳过其他层
**置信度**：高
**来源任务**：框架初始化
---

## [PAT-032] Master-Slave 明确原则
**适用场景**：机器人 + PLC 协同系统
**判断规则**：设计前必须明确谁是 Master（发出命令）谁是 Slave（执行并回复状态）；通常 PLC 为 Master，Robot 为 Slave；视觉系统由 Robot 或 PLC 触发；状态机在 Master 侧实现
**例外条件**：Robot 直接控制视觉且无 PLC 时，Robot 兼任 Master
**置信度**：高
**来源任务**：框架初始化
---

## [PAT-033] 握手协议设计规则
**适用场景**：设备间命令/状态通信
**判断规则**：每个命令必须有对应的状态回复；设计四状态握手：
  ① Master发命令 → ② Slave执行中 → ③ Slave完成/错误 → ④ Master确认收到后清除命令
  禁止单向发命令不等回复
**例外条件**：广播类数据（如心跳、状态广播）不需要握手
**置信度**：高
**来源任务**：框架初始化
